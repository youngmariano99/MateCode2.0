# Modelo Relacional

Versión: 2.0

> "Antes de crear una tabla debemos entender el negocio."

---

# Objetivo

Este documento define el modelo relacional general de MateCode.

No define columnas.

No define tipos de datos.

No define índices.

Su objetivo es mostrar las entidades del sistema y cómo se relacionan entre sí.

Cada entidad tendrá posteriormente su propio documento detallado.

---

# Filosofía

MateCode está construido como un conjunto de módulos independientes.

Cada módulo representa un área del negocio.

Las relaciones entre módulos deben ser claras, simples y fácilmente escalables.

Ningún módulo deberá depender innecesariamente de otro.

# Núcleo del Sistema

Estas entidades forman el corazón de MateCode.

## Agencia

Representa una empresa que utiliza MateCode.

Toda la información del sistema pertenece a una Agencia.

Ninguna entidad funcional puede existir sin una Agencia.

---

## Usuario

Representa una persona que utiliza MateCode.

Un usuario puede pertenecer a una o varias agencias (preparado para el futuro).

Cada usuario tendrá permisos y roles.

---

## Equipo

Permite organizar los miembros de una agencia.

No representa autenticación.

Representa organización interna.

---

# Branding

Cada Agencia posee un Branding.

Incluye:

- Logo
- Colores
- Design.md
- Datos fiscales
- Firma
- Información comercial

---

# Clientes

Representa un cliente confirmado.

Un Cliente puede tener:

- múltiples contactos
- múltiples proyectos
- múltiples pagos
- múltiples contratos
- múltiples direcciones
- múltiples notas
- múltiples etiquetas

Nunca almacenar toda esa información dentro de una única tabla.

---

# Contactos

Representan personas.

No empresas.

Un Cliente puede tener muchos contactos.

Ejemplo.

Empresa:

Panadería San Martín

Contactos:

Juan

Pedro

María

Cada contacto podrá tener:

- teléfono
- correo
- cargo
- observaciones

---

# Contactos en Frío

Representan negocios que todavía NO son clientes.

Su objetivo es gestionar prospección.

Podrán convertirse fácilmente en Clientes.

Nunca duplicar información.

La conversión deberá reutilizar los datos existentes.

---

# Direcciones

Las direcciones son entidades independientes.

Podrán utilizarse para:

Clientes

Contactos

Logística

Facturación

Visitas

Cada dirección podrá geolocalizarse.

---

# Logística

Representa recorridos.

No almacena direcciones.

Utiliza Direcciones.

Permitirá:

Optimizar rutas.

Agrupar visitas.

Registrar recorridos.

---

# Proyectos

Un Cliente podrá tener múltiples proyectos.

Cada proyecto podrá tener:

estado

fecha

responsable

archivos

comentarios

---

# Contratos

Un Cliente podrá tener múltiples contratos.

Cada contrato podrá:

editarse

versionarse

exportarse

firmarse

---

# Pagos

Un Cliente podrá tener múltiples pagos.

Los pagos podrán generar:

recordatorios

vencimientos

historial

estadísticas

---

# Historial

Toda acción importante generará un registro.

Ejemplo.

Cliente creado.

Cliente editado.

Proyecto iniciado.

Contrato enviado.

Pago registrado.

Nunca perder información.

---

# Actividad

La actividad representa acciones visibles para el usuario.

Ejemplo.

Kevin creó un cliente.

Mariano editó un proyecto.

Lucas registró un pago.

Alimenta el Dashboard.

---

# IA

La IA será un módulo independiente.

Nunca dependerá directamente de Clientes.

Se comunicará mediante interfaces.

Permitirá:

resumir

analizar

buscar

redactar

proponer

automatizar

---

# Automatizaciones

Representan reglas creadas por el usuario.

Ejemplo.

Cliente aceptado

↓

Crear Proyecto

↓

Crear Contrato

↓

Crear Pago

↓

Enviar Notificación

Nunca deberán depender de módulos específicos.

---

# Configuración

Representa preferencias generales.

Ejemplos.

Tema.

Idioma.

Notificaciones.

Integraciones.

---

# Logs

Representan eventos técnicos.

No visibles para el usuario.

Ejemplos.

ERROR

WARNING

INFO

BUG

Nunca mezclar Logs con Historial.

---

# Notificaciones

Representan avisos para el usuario.

Ejemplo.

Pago próximo.

Contrato pendiente.

Nuevo cliente.

# Escalabilidad

Nunca modificar entidades centrales para agregar funcionalidades nuevas.

Siempre extender.

Nunca romper.
